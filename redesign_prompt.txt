Texas Trace 프로젝트의 1차 구현 결과가 사용자의 의도와 크게 다릅니다.
따라서 기존 스크롤 기반 설계를 폐기하고, 아래 방향으로 설계 문서를 전면 수정해주세요.

중요:
- 지금은 코드 구현을 바로 수정하지 마세요.
- 먼저 PRD.md, TRD.md, TASKS.md, AGENT.md를 새 방향에 맞게 전면 개정하세요.
- 기존의 “한 페이지 스크롤 + sticky map + 많은 카드/정보” 구조는 폐기합니다.
- 새로운 방향은 “PPT를 넘기듯이 Next 버튼으로 여정 포인트를 하나씩 진행하는 인터랙티브 지도 프레젠테이션 웹”입니다.
- 화면에 이것저것 많은 정보를 동시에 보여주는 웹이 아닙니다.
- 최소한의 정보, 강한 지도/사진 경험, 포인트 간 이동 애니메이션이 핵심입니다.

# 1. 기존 설계에서 폐기할 것

다음 설계는 폐기하거나 후순위로 밀어야 합니다.

- 한 페이지 스크롤형 여정
- sticky map + 오른쪽에 긴 journey card list
- IntersectionObserver 기반 activeStep
- 스크롤하면서 포인트가 바뀌는 방식
- 많은 텍스트 카드와 정보가 한 화면에 계속 노출되는 구조
- 전체 장소를 한 번에 탐색하는 대시보드형 UI
- 여러 섹션을 길게 내려가며 읽는 블로그형 구조

사용자가 원하는 것은 이런 형태가 아닙니다.

# 2. 새 프로젝트 방향

새 방향:
- PC 발표용 “지도 기반 프레젠테이션 웹”
- 사용자가 직접 Next / Prev 버튼으로 포인트를 넘김
- PPT 슬라이드를 넘기듯이 1번 포인트 → 2번 포인트 → 3번 포인트 순서로 진행
- 각 포인트마다:
  - 지도는 해당 위치로 이동
  - 현재 포인트 마커가 강조
  - 이전 포인트에서 현재 포인트로 이동하는 선/trace가 애니메이션처럼 표현
  - 최소한의 제목/설명/사진만 표시
- 화면은 심플하고 몰입감 있어야 함
- 정보량보다 “이동하는 느낌”과 “여정의 흐름”이 중요함

한 문장 정의:
Texas Trace는 스크롤형 여행 블로그가 아니라, Next 버튼으로 여정을 따라가며 지도 위에서 텍사스 이동 궤적을 경험하는 프레젠테이션형 웹이다.

# 3. 새 UX 개념

화면은 기본적으로 하나의 “stage”처럼 구성합니다.

권장 레이아웃:

```text
┌──────────────────────────────────────────────────────────────┐
│ Texas Trace                                      03 / 19      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│                     Fullscreen / Large Map                   │
│                                                              │
│        현재 포인트 마커                                      │
│        이전 → 현재 이동 trace animation                      │
│                                                              │
├───────────────────────────────┬──────────────────────────────┤
│ Point Title                    │ Photo Strip / Photo Preview  │
│ 짧은 설명 1~3줄               │ 대표 사진 1~3장              │
│ role / route label             │ 클릭 시 전체화면             │
├───────────────────────────────┴──────────────────────────────┤
│ [Prev]                 Progress Dots / Timeline       [Next] │
└──────────────────────────────────────────────────────────────┘
```

또는:

```text
전체 배경: 지도
하단: 반투명 정보 패널
우측: 사진 미리보기
좌우: Prev / Next
상단: 현재 step 번호와 전체 진행률
```

중요:
- 화면은 발표자가 한 단계씩 넘기기 좋아야 합니다.
- 텍스트는 최소화합니다.
- 한 포인트당 핵심 문장 1~3줄 정도면 충분합니다.
- 사진은 대표 이미지 중심으로 보여줍니다.
- 장소별 모든 정보는 상세하게 펼치기보다, 필요하면 클릭/확대에서만 봅니다.

# 4. Next / Prev 기반 상태 모델

새 핵심 상태는 `currentStepIndex` 또는 `currentStepId`입니다.

기본 흐름:
1. 페이지 로드
2. stepId 1 표시
3. 사용자가 Next 클릭
4. stepId 2로 변경
5. 지도는 step 1 위치에서 step 2 위치로 이동
6. step 1 → step 2 route trace 강조/애니메이션
7. step 2의 제목/짧은 설명/사진 표시
8. 다시 Next 클릭 시 stepId 3으로 진행

Prev도 반대로 동작해야 합니다.

필수 컨트롤:
- Next 버튼
- Prev 버튼
- 현재 단계 표시: 예) 03 / 19
- progress dots 또는 간단한 horizontal timeline
- 특정 dot 클릭 시 해당 step으로 이동 가능하면 좋음
- 키보드 방향키 지원은 있으면 좋지만 초기 필수는 아님

# 5. 지도와 애니메이션 설계

지도는 화면의 중심입니다.

기본 지도:
- Leaflet 사용
- 기존 data/texasTraceData.js의 locations / journeySteps / routeLinks 사용
- routeLinks는 그대로 중요함
- journeySteps 순서대로 무조건 선을 자동 생성하면 안 됨
- 단, “현재 step으로 이동할 때 어떤 routeLink를 강조할지”는 routeLinks에서 찾아야 함

지도 표시 기본:
- 모든 포인트를 처음부터 강하게 다 보여주지 않아도 됨
- 현재 포인트는 가장 강하게 표시
- 이전까지 방문한 포인트는 은은하게 표시
- 아직 방문하지 않은 포인트는 약하게 표시하거나 숨겨도 됨
- 전체 route graph는 기본적으로 은은하게 깔고, 현재 이동 구간만 강조하는 방식 추천

애니메이션 우선순위:
1. step 변경 시 지도 중심 이동
2. 현재 마커 강조
3. 이전 step → 현재 step 연결선 강조
4. 이동 trace가 그려지는 듯한 간단한 line animation
5. 비행 구간과 자동차 구간의 스타일 구분

초기 구현에서 복잡한 3D 애니메이션은 금지.
부드럽고 안정적인 2D 지도 이동/선 강조/마커 pulse 정도로 충분함.

애니메이션 톤:
- 과하게 화려한 효과 금지
- 차분하고 고급스러운 프레젠테이션 느낌
- 선이 천천히 빛나거나 그려지는 정도
- 비행 구간은 dashed arc 또는 부드러운 긴 선
- local drive는 solid line
- long drive는 더 강조된 warm line
- 현재 포인트는 subtle pulse

# 6. routeLinks 사용 방식

routeLinks는 기존처럼 중요합니다.

절대 하지 말 것:
- 1 → 2 → 3 → 4 → ... → 19를 자동 연결해서 전체 지도 선을 만들지 말 것
- routeLinks를 무시하지 말 것
- 위치/좌표/연결 데이터를 UI 코드에 하드코딩하지 말 것

해야 할 것:
- data/texasTraceData.js의 routeLinks를 읽어서 지도 연결선을 만든다.
- 현재 step으로 이동할 때, 이전 step과 현재 step에 해당하는 routeLink를 찾는다.
- 해당 routeLink가 있으면 그 선을 강조/애니메이션 처리한다.
- 해당 routeLink가 없으면 fallback으로 지도 이동만 수행하고, 선 애니메이션은 생략한다.
- Houston 구간, Austin local hub 구간, 귀국 flight 구간의 의미가 유지되어야 한다.

# 8. 사진 표시 방식 변경

기존처럼 갤러리를 크게 아래에 펼치는 구조는 후순위로 둡니다.

새 방향:
- 각 step에서 대표 사진 1~3장을 보여줌
- 대표 장소는 3~5장까지 preview 가능
- 사진을 클릭하면 fullscreen modal에서 full 이미지 표시
- 별도 전체 갤러리 섹션은 초기 범위에서 제외하거나 매우 작게 둠
- 사진은 포인트별 경험을 보조하는 역할
- 지도/이동이 주인공이고, 사진은 현재 포인트의 감정 보강 요소

성능 원칙:
- thumb가 있으면 thumb 우선
- thumb가 없으면 full fallback
- modal에서는 full 이미지
- full 이미지 전체 preload 금지
- Next로 이동해도 다음 포인트 full 이미지를 대량 preload하지 말 것
- 필요하면 현재/다음 step의 thumb 정도만 가볍게 준비

# 9. 정보량 정책

사용자가 원하는 것은 “많은 정보”가 아닙니다.

각 step에서 보여줄 정보:
- step number
- title
- subtitle
- short summary 1~3줄
- role badge
- route label
- 대표 사진

표시하지 않아도 되는 것:
- 긴 설명
- 많은 카드
- 모든 장소 목록
- 긴 텍스트 블록
- 지도 옆에 빽빽한 정보
- 스크롤형 상세 글

상세 설명이 필요하다면 나중에 “More” 버튼으로 확장하는 구조를 고려할 수 있지만, 초기 구현에서는 최소 정보만 사용합니다.

# 16. 최종 목표 재확인

사용자가 원하는 Texas Trace는 다음과 같습니다.

- 화면에 많은 정보를 나열하는 웹이 아니다.
- 스크롤로 긴 페이지를 읽는 웹이 아니다.
- 지도와 정보 카드가 빽빽한 대시보드가 아니다.
- PPT를 넘기듯이 Next로 하나씩 진행한다.
- 각 포인트는 최소 정보와 사진만 보여준다.
- 포인트 간 이동에는 지도 애니메이션/trace가 적용된다.
- 지도 위에서 여정이 진행되는 느낌이 핵심이다.
- PC 발표용으로 시각적으로 강하고 간결해야 한다.
