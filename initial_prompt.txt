너는 이제 `texas-yoon` GitHub Pages 정적 웹 프로젝트의 개발 에이전트다.

목표는 2026년 5월 텍사스 출장/여행 경험을 `Texas Trace`라는 PC 발표용 인터랙티브 웹으로 구현하는 것이다.

중요:
- 지금은 코드 구현을 바로 시작하지 말 것.
- 먼저 프로젝트 문서와 데이터 설계를 안정적으로 작성하는 것이 1차 목표다.
- 반드시 아래 요구사항을 기반으로 `PRD.md`, `TRD.md`, `TASKS.md`, `AGENT.md`를 생성하라.
- 그 다음 `data/texasTraceData.js` 초안을 생성하라.
- `index.html`, CSS, JS 구현은 문서 생성 및 데이터 초안 생성 이후 별도 요청이 있을 때 시작하라.
- 사용자는 콘텐츠 작성자이고, Codex/AI는 구현 담당자다.
- 사용자가 나중에 쉽게 수정할 수 있도록 데이터 구조를 명확히 분리해야 한다.

# 2. PRD.md 작성 요구

`PRD.md`에는 다음 내용을 포함하라.

## 제품 목적
- `Texas Trace`는 2026년 5월 Austin / Houston / Texas 출장 및 여행 경험을 하나의 체험형 웹으로 보여주는 프로젝트다.
- 단순한 PPT나 Confluence 문서가 아니라, 지도, 여정, 사진, 스토리를 통해 웹 자체에서 경험을 전달한다.
- 발표자는 PC 화면에서 스크롤하며 여정의 흐름을 보여줄 수 있어야 한다.

## 사용자 경험 목표
- 사용자가 웹 안에서 다음 흐름을 자연스럽게 따라가야 한다.
  - 인천 출발
  - 애틀란타 환승
  - 오스틴 도착
  - 오스틴 숙소 중심 로컬 이동
  - 휴스턴 장거리 이동
  - NASA / 박물관 / Buc-ee’s 경험
  - UT Austin / COTA / Capitol 등 오스틴 경험
  - 오스틴 출국
  - 미니애폴리스 환승
  - 인천 도착
- 지도와 스토리가 연결되어야 한다.
- 사진은 감정과 기억을 전달하는 핵심 콘텐츠다.

## 주요 기능
- Hero / Entry 화면
- Trace Story 구간
- Leaflet 지도
- 19개 여정 포인트 마커
- 포인트 간 연결선

## 제외 범위
- 동영상
- 서버/백엔드
- 로그인
- 데이터베이스
- 모바일 최적화

## 성공 기준
- GitHub Pages에서 정상 배포된다.
- 19개 포인트가 정확한 좌표에 표시된다.

- 사진 클릭 시 full 이미지를 전체화면으로 보여준다.

## 데이터 구조 원칙

가장 중요한 설계 원칙:
- 여정 포인트 데이터
- 실제 장소 데이터
- 포인트 간 연결 데이터
- role / 의미 데이터
- 사진 데이터

위 데이터를 절대 한 덩어리로 섞지 말고 분리한다.

`data/texasTraceData.js`는 전역 객체 방식으로 작성한다.
빌드 도구 없이 GitHub Pages에서 바로 사용할 수 있어야 하므로 `export/import`보다 아래 형태를 우선한다.

```js
window.TEXAS_TRACE_DATA = {
  meta: {},
  locations: {},
  journeySteps: [],
  routeLinks: [],
  mediaGroups: {}
};
```

## locations
실제 장소 자체를 나타낸다.
같은 장소는 한 번만 정의한다.

예:
- 인천공항은 step 1, step 19에서 재사용된다.
- 오스틴 공항은 step 3, step 17에서 재사용된다.

필드 예시:
```js
"incheon-airport": {
  name: "인천공항",
  officialName: "Incheon International Airport",
  lat: 37.463333,
  lng: 126.440002,
  address: "Incheon International Airport, Incheon, South Korea",
  category: "airport",
  country: "KR"
}
```

## journeySteps
시간 순서상 여정 포인트다.
1번부터 19번까지 화면 스토리 순서로 표시한다.

필드 예시:
```js
{
  stepId: 1,
  locationId: "incheon-airport",
  title: "인천공항",
  subtitle: "텍사스 여정의 시작",
  role: "departure",
  storyType: "flight-start",
  summary: "텍사스 출장/여행의 시작점.",
  mediaGroupId: "incheon-airport-departure"
}
```

## routeLinks
지도에서 실제로 연결선을 그릴 데이터다.

중요:
- `journeySteps`의 순서를 기준으로 자동 연결하면 안 된다.
- 지도 연결선은 반드시 `routeLinks`만 기준으로 그린다.
- 실제 이동 스토리 기준으로 routeLinks를 구성한다.
- 초기 구현에서는 routeLinks 선 표시와 강조만 구현한다.
- 애니메이션은 후순위다.

## 스크롤 + sticky map 구조

초기 레이아웃은 다음 방향으로 설계한다.

- Hero 구간은 전체 인트로
- Trace Story 구간에서 지도는 sticky 형태로 유지
- 스토리 카드 영역은 시간 순서로 스크롤
- Gallery 구간에서는 지도 sticky를 해제하고 사진 중심 레이아웃
- Final Note는 감성적 마무리

개념:
```text
Hero
↓
Trace Story Section
  - 왼쪽: sticky Leaflet map
  - 오른쪽: journey step cards
↓
Photo Gallery
↓
Final Note
```

activeStep 설계:
- 각 journey step section/card는 `data-step-id`를 가진다.
- `IntersectionObserver`를 사용해 현재 화면에 들어온 step을 activeStep으로 설정한다.
- activeStep이 바뀌면:
  - 해당 마커 강조
  - 관련 routeLinks 강조
  - 해당 스토리 카드 강조
  - 필요 시 지도 중심 이동(flyTo)은 후순위 또는 최소 기능으로 구현
- 초기 구현에서 복잡한 경로 이동 애니메이션은 제외한다.
- 나중에 activeStep 변경 이벤트를 기반으로 flyTo, line drawing, marker pulse animation을 추가할 수 있게 구조화한다.

## 이미지 성능 설계

성능 정책은 최우선으로 고려한다.

사진 정책:
- 사용자는 기본적으로 `full` 폴더에 고화질 이미지만 넣어도 웹이 동작해야 한다.
- `thumb` 폴더에 같은 파일명의 썸네일이 있으면 초기 갤러리에서는 thumb를 사용한다.
- thumb가 없거나 로딩 실패하면 full 이미지를 fallback으로 사용한다.
- 사진 클릭 시 전체화면 모달에서는 항상 full 이미지를 사용한다.
- 전체화면을 열기 전에는 full 이미지를 preload하지 않는다.
- 모든 full 이미지를 한 번에 로드하지 않는다.
- `loading="lazy"`를 적극 사용한다.
- Hero 대표 이미지는 1장만 사용하고, 지나치게 큰 원본을 피한다.
- 동영상은 초기 범위에서 제외한다.

이미지 폴더 규칙:
```text
assets/images/places/{place-slug}/full/01.jpg
assets/images/places/{place-slug}/thumb/01.jpg
```

데이터에는 `basePath`와 `file`만 적는다.

예:
```js
mediaGroups: {
  "space-center-houston": {
    basePath: "assets/images/places/space-center-houston",
    images: [
      {
        file: "01.jpg",
        caption: "Space Center Houston"
      }
    ]
  }
}
```

코드가 자동으로 다음 경로를 만든다.
```text
thumb: {basePath}/thumb/{file}
full: {basePath}/full/{file}
```

초기에는 thumb가 없어도 full fallback으로 동작해야 한다.

추후 개선:
- `tools/generate-thumbnails.js` 또는 `tools/generate_thumbnails.py`로 full 이미지를 기반으로 thumb 이미지를 자동 생성할 수 있게 한다.
- 단, 썸네일 생성 스크립트는 초기 필수 구현 범위가 아니다.

사진 개수 정책:
- 일반 장소: 1~3장
- 중요 장소: 3~5장
- 대표 장소: 10장 내외 가능
- 전체 초기 권장: 40~60장 이하
- 사진이 많아도 최초 로딩에서 full 이미지를 대량 로드하지 않도록 설계한다.

## GitHub Pages 성능 정책

성능 정책:
- 저장소와 published site 크기를 작게 유지한다.
- 고화질 사진이 가장 큰 성능 병목임을 전제로 설계한다.
- 초기 렌더링에서는 thumb 우선, full fallback
- 전체화면 모달을 열 때만 full 이미지 사용
- 모든 full 이미지 preload 금지
- 스크롤 activeStep은 IntersectionObserver 사용
- scroll 이벤트에 무거운 연산을 직접 연결하지 않는다.
- 지도 마커는 19개이므로 성능 문제는 작지만, 불필요한 재렌더링은 피한다.
- 외부 지도 타일 서버는 하나를 기본으로 사용하고, 여러 tile layer를 동시에 로드하지 않는다.

# 5. AGENT.md 작성 요구

`AGENT.md`에는 개발 에이전트 지침을 작성하라.

반드시 포함할 내용:

## 개발 원칙
- 사용자는 콘텐츠 작성자이고, AI가 구현 담당자다.
- 사용자가 데이터 파일을 쉽게 수정할 수 있어야 한다.
- 장소/여정/연결/role/사진 데이터를 섞지 않는다.
- routeLinks는 journeySteps와 별개다.
- 지도 연결선은 routeLinks만 기준으로 그린다.
- 동일 locationId가 여러 stepId에서 재사용될 수 있다.
- 초기 구현에서 동영상과 복잡한 애니메이션은 제외한다.
- 불필요한 프레임워크를 도입하지 않는다.
- HTML/CSS/Vanilla JS로 시작한다.
- GitHub Pages에서 바로 동작해야 한다.

## 변경 관리
- 한 번에 너무 많은 기능을 구현하지 않는다.
- TASKS.md의 순서를 따른다.
- 각 단계 완료 후 사용자가 브라우저에서 확인할 수 있는 상태를 유지한다.
- 기존 기능을 깨지 않도록 작은 단위로 변경한다.
- 임의로 데이터 구조를 단순화하지 않는다.
- 임의로 routeLinks를 자동 생성하지 않는다.
- 임의로 모바일 최적화를 우선하지 않는다.

## 좌표 원칙
- 좌표는 사용자 경험에 매우 중요하다.
- 임의 추정 좌표를 사용하지 않는다.
- 아래 제공된 좌표 seed를 사용하되, 구현 전 가능한 한 주소/공식 장소명 기반으로 한 번 더 검토한다.
- 런타임에서 geocoding API를 호출하지 않는다.
- 확정된 lat/lng 값을 data 파일에 저장한다.
- 좌표가 불확실한 경우 data에 `coordinateStatus: "needs-review"`를 명시한다.

## 이미지 원칙
- full 이미지만 있어도 웹이 동작해야 한다.
- thumb가 있으면 초기 로딩에서 thumb를 사용한다.
- thumb가 없으면 full fallback을 사용한다.
- 사진 클릭 시 전체화면에서는 full 이미지를 사용한다.
- full 이미지를 전체 페이지 초기 로딩 시 preload하지 않는다.
- 이미지가 없어도 전체 페이지가 깨지지 않게 한다.
- 동영상은 초기 범위에서 제외한다.

## 성능 원칙
- 성능을 최우선으로 고려한다.
- 고화질 사진이 가장 큰 병목이라는 전제로 설계한다.
- lazy loading을 적용한다.
- IntersectionObserver를 사용한다.
- scroll 이벤트에 무거운 작업을 직접 연결하지 않는다.
- 지도 tile layer를 여러 개 동시에 로드하지 않는다.
- 전체 페이지 초기 로딩을 가볍게 유지한다.
